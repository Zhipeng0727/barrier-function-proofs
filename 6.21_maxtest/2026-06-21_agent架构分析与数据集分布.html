② 结构性问题(按严重度)

🟢 1. Refuter gate 方向反了 — ✓ 已修复(2026-06-22)
原来干净多项式被 gate 自动接受、跳过检查。已改为路由到 sound 后端:sound_arbiter.sound_verify 把 poly/rational 交给 verif_agent 的 Z3 后端严格判定;z3 proved ⇒ verify_level=L2_smt(sound)、solved=True,且 LLM Refuter 完全不介入(实测 refuter_calls=0)。
🟢 2. 用 LLM 当验收 oracle 不 sound — ✓ 已修复
LLM Refuter "valid" 不再单独置 solved。它降级为非权威候选标记:verify_level=L1_llm、candidate=True、solved 保持 False。只有 Z3(L2_smt)或 Lean(L3_lean)这类严格形式化验证才能令 solved=True。
🟢 3. 网络故障被升级成成功 — ✓ 已修复
except RuntimeError 路径不再置 solved。改为 verify_level=error_network、candidate=True、solved=False 的独立状态,benchmark 不计入 solved。实测确认。
🟠 4. cond2 的 band+CBF 松弛是好过滤、坏终判
在 0≤h<ε band 采样,对 C∈{0,1,2,5,10,25,50} 试 ḣ+C·h≥−noise,任一过且"违反<1%或不深"即过。会放过小测度真泄漏;且把被认证性质从 Nagumo 悄悄换成 exponential-CBF,solved 含义未记录。
🟠 5. Lean 那条 sound 路也有洞
生成证明曾用 axiom nagumo_barrier_invariance 公理化微积分桥,甚至自造 axiom Real 假实数(见 v4_darboux.lean 的 12 条公理);旧 lean_compile_check 只查"编译+无sorry",放过这类假证明。
✓ 2026-06-22 已修复：① 三级升级 formalize_certificate 接入主循环(图2);② L4 soundness 审计 lean_soundness.audit 内嵌 lean_compile_check —— 静态禁 axiom/admit + #print axioms 验证只依赖 {propext,Classical.choice,Quot.sound} + 陈述非平凡 lint。axiom-cheat / 自造ℝ / : True 三类假证明实测均判 ok=False。 桥若真无法证,只能升 L3 换 formalization-friendly 证书,禁止公理化造假。
🟡 6. 收敛:Refuter 无状态 + 紧凑 critique → 可能震荡
反例缓存只挡 Cond1 重复,挡不住 Cond2/Refuter-flaw 重复提议。唯一防卡死是 effort ladder,无"禁止重提已驳回证书"去重。
🟡 7. 没区分 Refuter 真 bug vs 幻觉 bug
误判(把对的判 invalid)让 Proposer 追不存在的 bug,白烧轮数。无交叉校验(如把声称的反例丢回 local_verify 确认)。
🟢 8. 全局可变状态挡并行
TOKENS / API_CONFIG 进程级可变 dict;core200 批量并行会 race,限制吞吐。
