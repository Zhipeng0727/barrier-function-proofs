关键机制：本地验证是一道“门”
Refuter（贵的 LLM 调用）只有在本地验证 passed=True 时才会被触发。所以本地验证过严 → 候选永远进不了 refuter → refuter_calls=0 且永远不收敛。这正是 claude 两道题卡死的直接机制。

② 提示词结构（两个 Agent 各自的 system prompt + 消息组装）

两个 Agent 共享同一套数学框架（命题4 / Nagumo 定理），区别在角色：Proposer 构造、Refuter 证伪。下面是真实源码 dual_agent_debate_3/4.py 中的提示词骨架。

📐 共享的数学框架（两个 prompt 都注入）

寻找 barrier h(x)，使超水平集 C = {x∈X : h(x) ≥ 0} 为不变集（只要不进 unsafe 集即可，不要求严格留在 X 内）：
Condition 1   h(x) < 0,   ∀x ∈ X_u  (unsafe 集落在 h 的负区) Condition 2   ḣ(x) ≥ 0,   ∀x ∈ ∂C  (边界 h=0 上的 Nagumo 条件)
其中 ḣ(x) = ∇h(x)·f(x) 是 Lie 导数。每个 prompt 都注入具体的 System name / Dynamics / X_domain / X_u / 状态变量。
PROPOSER · system 🧠 控制论专家：构造 barrier
角色设定
"You are a control theory expert specializing in barrier functions and invariant sets." 目标：让不变集 C 尽可能大；h 不必是多项式，可由定理证明反推构造。
构造提示
优先解析表达式（三角/多项式/指数等），避免轨迹、向量场这类难直接验证的对象。Cross-term 验证须严格完整。
⚠ 对 "h" 字段的严格约束（喂给本地 sympy 验证用）
① 只含状态变量和纯数字常数，禁止 delta/epsilon 等符号参数；② 小正常数写成显式数值(如 0.01)；③ 用 x**2 不用 x^2；④ 允许函数 exp/sin/cos/tan/log/sqrt/Abs；⑤ 禁止中间变量名(A,P,F,V,φ)，h 必须完全展开成状态变量。
强制输出 JSON 格式
{
  "h": "h(x) 表达式, e.g. 1 - x1**2 - x2**2",
  "invariant_set": "C 的描述",
  "condition1_check": "验证 h<0 在 X_u 上成立的推导",
  "lie_derivative": "ḣ = ∇h·f 的计算过程",
  "condition2_check": "验证 ḣ≥0 在 ∂C(h=0) 上的推导",
  "reasoning": "选择该 h 的理由"
}
PROPOSER · user msg 🧩 每轮动态组装（build_proposer_msg）
1
基础任务 — System / X_domain / X_u，要求 C 尽可能大
2
KB few-shot — 仅第 1 轮注入知识库范例 (barrier_kb.json)
3
历史失败 — 最近 4 次 h 及失败原因，"不要重蹈覆辙"
4
已知反例点 — cache 里的 X_u 点 / 逃逸点，新 h 必须全部处理
5
Refuter 批评 — 上一轮验证器的 critique(截断 1500 字)
6
扩大提示 — 若上轮 h 已 valid，给出扩大 C 的建议(可选)
REFUTER · system 🔎 验证专家：找反例
角色设定
"a rigorous mathematical verification expert specializing in finding flaws and counterexamples." user msg 注入 System / X_u / 待验证的 h。
验证步骤
① 选 X_u 内具体点验 Condition 1（先数值后形式化不等式）→ ② 重算 Lie 导数 ḣ=∇h·f → ③ 在边界 ∂C(h=0) 上验 Condition 2(Nagumo) → ④ cross-term 严格证明 → ⑤ (仅当 valid) 分析能否扩大 C。
强制输出 JSON 格式
{
  "verdict": "valid" | "invalid",
  "condition1_result": "带具体数值的验证结果",
  "lie_derivative_computed": "重算的 ∇h·f",
  "condition2_result": "边界 Nagumo 条件验证结果",
  "counterexample": "若 invalid 给反例点+数值；valid 则 null",
  "flaw": "若 invalid 指出哪个条件失败",
  "suggestion": "若 invalid 给修正指引",
  "enlargement_suggestion": "若 valid 且可扩大，给方向"
}
⚡ LOCAL VERIFY · 非 LLM Proposer 与 Refuter 之间的硬门（纯 Python/sympy）
① precheck_cache
把 h 代入缓存的 X_u 点 / 逃逸点，快速查 Condition 1 是否被破坏。
② nontrivial_C_check  已于 20:04 加入
域内随机采 5000 点，要求 ≥5 点满足 h≥0，否则判 C 空集/退化（专门堵 h=-1 这类钻空子）。codex 那次 run(19:xx) 早于此补丁，故 X03 才得逞。
③ verify_symbolic  已改 (b)
符号求 ḣ，在 ε-边界采样数百~数千点：cond1_ok(X_u 上 h<0)、cond2_ok(边界 ḣ≥0)。旧：任一采样点 ḣ<0 即失败（单点否决，claude 进不了门的根因）。新：仅当违反比例 >1% 且深度显著(<−max(1e-3, 5%×边界|ḣ|中位数))才失败——孤立毛刺放行、系统性漏才杀；ḣ 爆炸也计入违反。复算：好解零误杀，claude X03 t1(违反仅0.46%)由 FAIL→PASS 放行进 refuter。
④ verify_trajectory
scipy 数值积分若干轨迹，查 C 内出发是否始终 h≥0。
四步全过 → passed=True → 才调用 Refuter（贵的 LLM）。任一步失败 → 反馈拼进下一轮 Proposer 消息，refuter 不触发。

⑤ 暴露的问题（待商量调整）

A. 本地验证过严 → claude 进不了门（X01 / X03 跑满 5 轮，refuter_calls=0） 已改 (b)
claude 每个候选 cond1 基本都过，但 cond2（边界 ḣ≥0）反复失败：min_lie 始终为负（−58、−40、甚至 −2e17）。旧判据单点否决对 exp/log 这类边界梯度巨大的候选极不友好。已改为"违反比例+深度"判据（(b)）：复算显示好解零误杀、X01(违反16~73%)等系统性漏仍被拒，仅 claude X03 t1(违反0.46%)放行进 refuter。
B. 空不变集漏洞 → codex 用 h=-1 钻空子（X03） 已打补丁
codex 先试 h=x3(cond2 −0.098 被拒)，随后论证“严格条件下不存在非空 barrier”，提交常数 h=-1：C={−1≥0}=空集 → 边界点数=0 → cond2 空泛成立，passed=True，refuter 也判 valid。但这是 codex run(19:xx) 时的情况；20:04 已加入 nontrivial_C_check（域内 ≥5 点须满足 h≥0），现在这类退化解会被直接拒绝。建议：补跑一次 codex 验证补丁是否真的堵住。
C. JSON 解析容错差 → 白烧轮次（两边都有，claude 尤甚）
Proposer 有时输出大段推理但 JSON 里只有 raw 字段、没有干净 h，解析器抽不出候选（图中 🟡）。claude X01/X03/X02 各有 1–2 轮这样空转，5 轮预算实际只剩 3 次有效尝试。
D. 工程崩溃 → codex X05 整题失败
PrintMethodNotImplementedError ... Derivative：sympy 表达式残留未求值的 Derivative，NumPyPrinter 生成代码时崩溃。与模型能力无关，同题 claude 1 轮即解。

⑥ 可以商量的调整方向

放宽 / 改进本地验证（针对 A）

  • cond2 引入容差 ε_tol：允许 min_lie ≥ −δ（少量采样噪声）而非严格 ≥0
  • 边界采样改为只罚“持续违反”：违反点占比 / 违反深度阈值，而非单点否决
  • 对 exp/log 候选归一化梯度或在 ḣ 上做尺度无关判据，避免 −2e17 这种数值爆炸主导
  • 把本地验证从“硬门”改成“软分”：低分也允许进 refuter，让 LLM 做最终裁决

堵空集漏洞（针对 B · 已部分完成）

  • ✓ 已加 nontrivial_C_check：域内 ≥5 点须满足 h≥0
  • 建议补：在 refuter 提示词里显式禁止 vacuous / 常数解（双保险）
  • 建议补：补跑 codex 确认补丁后 X03 不再被判 solved

解析容错（针对 C）

  • raw-only 输出时用正则/二次 LLM 回退抽取 h=
  • 解析失败不计入轮次预算，或当场重试一次

修崩溃（针对 D）

  • codegen 前 expr.doit() 求值掉 Derivative，或 NumPyPrinter(strict=False)
  • 给整题包 try/except，崩溃记为单题失败而非中断全 run
我的核心纠结点（请你拍板）
本地验证过松会放进一堆假候选、refuter 成本飙升、还可能像 codex 那样被钻空子；过严又像 claude 这样根本进不了门。建议方向：cond2 加容差 + 违反比例判据（治 A），同时加“C 非空”硬约束（治 B）——两者不冲突，一个放宽数值严格度、一个堵死退化解。你倾向硬门+容差，还是改成软分让 refuter 兜底？
